# متطلبات موديول المشاغل والتكاملات الخارجية

## 1. الهدف والنطاق

تهدف هذه الوثيقة إلى تلخيص متطلبات موديول المشاغل الجديد ضمن نظام Zaha-core-orbit، وتحديد المطلوب من الجهات الخارجية قبل التنفيذ، خصوصاً نظام إيجابي، موقع زها التجاري، وبوابة الدفع الإلكتروني.

يشمل نطاق المرحلة الحالية دورة الطلب كاملة من إنشاء الطلب وحتى الإنتاج، الاعتماد، عدم المطابقة، الكانبان، المعايرة، التقارير الزمنية، والتكاملات المطلوبة. لا يتم ترحيل أي جزء إلى مرحلة ثانية إلا إذا قررت الإدارة خلاف ذلك لاحقاً.

## 2. مصادر الطلبات

تصل طلبات المشاغل من مصدرين رئيسيين:

1. قسم العلاقات، من خلال شاشة إدخال داخل نظام Zaha-core-orbit.
2. مشتري مباشر من خلال Website الشوب، مع تمرير الطلب إلى نظام المشاغل عبر التكامل.

## 3. بيانات طلب المشاغل الأساسية

يجب أن تحتوي شاشة الطلب على الحقول التالية:

| الحقل | الوصف | المصدر المتوقع |
| --- | --- | --- |
| تاريخ الطلب | تاريخ زيارة الزبون أو تاريخ إنشاء الطلب | العلاقات / الموقع |
| رقم الطلبية | معرف الطلب الفريد | النظام / الموقع |
| اسم المشتري | الاسم الكامل للمشتري | العلاقات / الموقع |
| رقم هاتف المشتري | رقم التواصل | العلاقات / الموقع |
| اسم المنتج | اسم المنتج المطلوب | إيجابي / الموقع / إدخال يدوي |
| كود المنتج | كود المادة أو المنتج | إيجابي / الموقع |
| كمية المنتج | الكمية المطلوبة | العلاقات / الموقع |
| تغليف خاص | نعم / لا | العلاقات / الموقع |
| موعد التسليم | الموعد المطلوب من المشتري | العلاقات / الموقع |
| حالة المنتج | موجود أو جديد | العلاقات / الموقع / الموقع |

في البداية يتم اختيار المنتج من قائمة منسدلة للمنتجات المتوفرة، على أن تُغذّى هذه القائمة لاحقاً من إيجابي أو من ملف المنتجات الذي سيتم تزويده.

## 4. بيانات المنتج الجديد

إذا كان المنتج جديداً، يجب فتح قسم إضافي يحتوي على:

- صنف المنتج.
- اللون.
- القياس.
- صورة المنتج إن وجدت.
- مواصفات أخرى نصية.
- هل يرغب المشتري باستلام عينة؟
- موعد تسليم العينة إذا كانت الإجابة نعم.

## 5. مسارات الاعتماد والتشغيل

### 5.1 المنتج الجديد

1. ينتقل الطلب إلى لجنة المشاغل.
2. تقوم اللجنة بالمراجعة، الاعتماد، والتسعير.
3. بعد الاعتماد ينتقل الطلب إلى رئيس قسم المشاغل.
4. يقوم رئيس قسم المشاغل بالاعتماد النهائي أو طلب تعديل.
5. بعد الاعتماد ينتقل الطلب إلى موظفي قسم المشاغل للتنفيذ.

لجنة المشاغل مطلوبة فقط عند تفصيل منتج جديد كلياً.

### 5.2 المنتج الموجود

1. ينتقل الطلب إلى موظف الشوب.
2. يقوم موظف الشوب بتحديد الكمية المتوفرة.
3. إذا كانت الكمية متوفرة بالكامل ينتهي الطلب من ناحية التصنيع.
4. إذا كانت الكمية غير متوفرة أو متوفرة جزئياً ينتقل الطلب إلى رئيس قسم المشاغل.
5. يحدد رئيس قسم المشاغل الاعتماد، التعديل المطلوب، وتاريخ التسليم المتوقع.

### 5.3 التغليف الخاص

إذا كانت الطلبية بحاجة إلى تغليف خاص، يتم إرسال طلب أو مهمة إلى قسم الاتصالات بعد اعتماد رئيس قسم المشاغل.

## 6. موظفو المشاغل والإنتاج

يوجد أكثر من موظف مشاغل، وكل موظف مسؤول عن صنف محدد. تظهر الطلبات لجميع موظفي المشاغل، لكن القرار والتنفيذ يكون عند الموظف المسؤول عن صنف الطلب.

تظهر شاشة الإنتاج كجدول يحتوي على:

- الرقم التسلسلي للمنتج.
- صنف المنتج.
- الكمية المطلوبة.
- مراحل الإنتاج.

مراحل الإنتاج تختلف عن مراحل الاعتماد، ويجب أن تكون قابلة للتعديل أو الإعداد عند توفر التفاصيل النهائية.

## 7. مراحل الإنتاج وعدم المطابقة

لكل مرحلة إنتاج يجب تسجيل:

- المرحلة الحالية.
- نسبة الإنجاز أو حالة الإنجاز.
- الجهة أو الشخص المسؤول.
- تاريخ البداية وتاريخ الانتهاء عند الحاجة.

في كل مرحلة يجب توفير إمكانية إضافة حالة عدم مطابقة من نفس الشخص الذي يحدث مرحلة الإنتاج. عند إدخال عدم مطابقة يجب إيقاف العملية وإظهارها لرئيس قسم المشاغل والموظفين المعنيين.

### 7.1 حقول عدم المطابقة

| الحقل | الوصف |
| --- | --- |
| رقم الحالة | معرف حالة عدم المطابقة |
| رقم الطلب | الطلب المرتبط |
| صنف الطلب | صنف المنتج أو الطلب |
| كمية الطلب | الكمية المتأثرة |
| وصف عدم المطابقة | وصف نصي للحالة |
| تصنيف الحالة | قائمة اختيار سيتم تزويدها لاحقاً |
| تحليل الأسباب | تحليل سبب المشكلة |
| الإجراء التصحيحي | إجراء حل المشكلة جذرياً |
| مسؤول التنفيذ | الشخص المسؤول عن الإجراء التصحيحي |
| التاريخ المتوقع | موعد تنفيذ الإجراء التصحيحي |

## 8. الكانبان وتحليل سير العمل

### 8.1 البورد العام

يتم توفير Kanban عام بأسلوب Trello يحتوي على الأعمدة التالية:

- قيد الطلب / In Order.
- قيد التنفيذ / In Progress.
- منجز / Done.

تعرض المنتجات على شكل Cards، ويمكن لجميع مستخدمي المشاغل ومنسق الفروع مشاهدة البورد.

### 8.2 البورد التفصيلي

يتم توفير بورد تفصيلي تكون أعمدته معرفة من المستخدمين. تحتوي كل Card على:

- Checklist لمراحل التصنيع.
- تاريخ البدء.
- تاريخ الانتهاء.
- اسم الموظف المسؤول عن العملية.

جميع المعلومات في البورد التفصيلي نصية وقابلة للتعديل.

### 8.3 تحليل سير العمل

يجب توفير Flow أو Timeline يوضح:

- موقع الطلبية الحالي.
- الزمن الذي استغرقته في كل مرحلة.
- إبراز المرحلة التي استغرقت أكبر وقت.

## 9. المعايرة والصيانة

### 9.1 بيانات الجهاز

يدخل الأدمن بيانات الأجهزة التالية:

- الشركة المصنعة.
- تاريخ الإنشاء.
- المسؤول عن الجهاز.
- حالة الجهاز.

### 9.2 خطة المعايرة

يدخل الأدمن خطة المعايرة من خلال تقويم يحدد:

- مواعيد معايرة كل جهاز.
- جدول المعايرات المستقبلية.

### 9.3 متابعة المعايرة والصيانة

يقوم رئيس قسم المشاغل بإدخال:

- هل تمت الصيانة أم لا.
- تاريخ الصيانة.
- منفذ الصيانة.
- حالة الجهاز بعد الصيانة.

## 10. الصلاحيات

يجب دعم الصلاحيات التالية كحد أدنى:

| الدور | الصلاحيات الرئيسية |
| --- | --- |
| منسق الفروع | إنشاء طلبات العلاقات، متابعة البورد العام، استلام طلبات التعديل |
| موظف الشوب | مراجعة طلبات المنتجات الموجودة وتحديد الكمية المتوفرة |
| لجنة المشاغل | اعتماد وتسعير المنتجات الجديدة |
| رئيس قسم المشاغل | الاعتماد النهائي، طلب التعديل، تحديد تاريخ التسليم، متابعة عدم المطابقة والمعايرة |
| موظف المشاغل | متابعة وتنفيذ الطلبات حسب الصنف، تحديث مراحل الإنتاج، إدخال عدم المطابقة |
| قسم الاتصالات | متابعة مهام التغليف الخاص |
| ضبط الجودة | صلاحية View على جميع بيانات وعمليات النظام |
| الأدمن | إعداد الأجهزة، خطة المعايرة، القوائم والتصنيفات |

## 11. المطلوب من نظام إيجابي

يجب مخاطبة فريق إيجابي لتوفير واجهات قراءة فقط للبيانات التالية، لأن إيجابي هو قاعدة البيانات الرئيسية لبيانات المواد والمستودعات والأسعار الخاصة بمنتجات زها التجارية والشوب.

### 11.1 APIs مطلوبة من إيجابي

| API | الغرض | نوع الوصول |
| --- | --- | --- |
| قائمة المواد | جلب قائمة المنتجات والمواد المعروضة في الشوب | Read-only |
| تفاصيل مادة | جلب بيانات مادة محددة حسب الكود | Read-only |
| توفر المخزون | معرفة الكمية المتوفرة حسب كود المادة والمستودع أو الفرع | Read-only |
| أسعار المواد | جلب سعر المادة والعملة والضريبة إن وجدت | Read-only |
| تصنيفات المواد | جلب الأصناف والفئات المستخدمة في الشوب | Read-only |
| صور أو مرفقات المادة | جلب روابط صور المنتج إن كانت متوفرة | Read-only |

### 11.2 الحقول المطلوبة من إيجابي

- كود المادة.
- اسم المادة بالعربي.
- اسم المادة بالإنجليزي إن وجد.
- وصف المادة.
- الصنف أو الفئة.
- السعر.
- العملة.
- نسبة أو قيمة الضريبة إن وجدت.
- حالة المادة: فعالة أو غير فعالة.
- الكمية المتوفرة.
- المستودع أو الفرع.
- وحدة القياس.
- الباركود إن وجد.
- رابط الصورة أو المرفقات إن وجدت.
- آخر تاريخ تحديث للبيانات.

### 11.3 أسئلة يجب تثبيتها مع إيجابي

1. ما نوع المصادقة المعتمد للـ API؟
2. هل البيانات متاحة عبر REST API أم Database View أم ملف دوري؟
3. ما بيئة الاختبار وبيئة الإنتاج؟
4. ما معدل التحديث المتوقع للمخزون والأسعار؟
5. هل الكمية المتوفرة تكون حسب مستودع واحد أم حسب عدة مستودعات؟
6. هل السعر شامل الضريبة أم غير شامل؟
7. هل يوجد API لحجز الكمية أم المطلوب قراءة فقط في هذه المرحلة؟
8. ما صيغة الأخطاء والرسائل المرجعة؟
9. هل يوجد Rate Limit أو قيود IP Whitelist؟
10. هل أكواد المواد ثابتة ولا تتغير؟

## 12. المطلوب من Website زها التجاري

يجب تحديد آلية إرسال الطلبات من الموقع إلى نظام Zaha-core-orbit عند وجود منتج يحتاج تصنيعاً أو طلب خاص.

### 12.1 APIs مطلوبة بين الموقع ونظام المشاغل

| API | الاتجاه | الغرض |
| --- | --- | --- |
| إنشاء طلب مشاغل | Website إلى Zaha-core-orbit | إرسال طلب شراء أو تصنيع جديد |
| تحديث حالة الطلب | Zaha-core-orbit إلى Website | إظهار حالة الطلب للمشتري |
| تحديث تاريخ التسليم المتوقع | Zaha-core-orbit إلى Website | إعلام الموقع بتاريخ التسليم |
| إشعار العينة | Zaha-core-orbit إلى Website | إرسال موعد تسليم العينة إن وجدت |
| إلغاء أو تعديل طلب | Website إلى Zaha-core-orbit | تمرير طلب تعديل أو إلغاء حسب السياسة |

### 12.2 بيانات الطلب المطلوبة من الموقع

- رقم طلب الموقع.
- رقم العميل إن وجد.
- اسم العميل.
- رقم الهاتف.
- البريد الإلكتروني إن وجد.
- عنوان التسليم إن وجد.
- كود المنتج.
- اسم المنتج.
- الكمية.
- السعر وقت الطلب.
- العملة.
- حالة الدفع.
- رقم عملية الدفع.
- هل الطلب يحتاج تصنيع؟
- هل الطلب يحتاج تغليف خاص؟
- ملاحظات العميل.
- روابط صور أو مرفقات للطلب الخاص إن وجدت.
- تاريخ إنشاء الطلب.

## 13. المطلوب من بوابة الدفع الإلكتروني

يجب طلب تفاصيل مخرجات الدفع الإلكتروني من الجهة المسؤولة عن الموقع أو بوابة الدفع، حتى يتم ربط الطلب المالي بطلب المشاغل عند الحاجة.

### 13.1 الحقول المطلوبة من الدفع الإلكتروني

- Payment ID.
- Transaction ID أو Gateway Reference.
- Order ID المرتبط بطلب الموقع.
- مبلغ العملية.
- العملة.
- حالة الدفع: Paid, Failed, Pending, Refunded, Partially Refunded.
- طريقة الدفع: Card, Wallet, Transfer أو غيرها.
- تاريخ ووقت الدفع.
- رمز التفويض Authorization Code إن وجد.
- آخر أربعة أرقام من البطاقة إن كان مسموحاً.
- اسم مزود خدمة الدفع.
- رسوم العملية إن وجدت.
- رابط أو رقم الفاتورة.
- بيانات الاسترجاع Refund إن وجدت.
- توقيع التحقق Webhook Signature.

### 13.2 Webhooks مطلوبة

- Payment succeeded.
- Payment failed.
- Payment pending.
- Payment refunded.
- Chargeback أو dispute إن كانت مدعومة.

## 14. الرد المقترح إرساله للجهات الخارجية

مرحباً،

لغايات تنفيذ موديول المشاغل وربطه مع الأنظمة الخارجية، نحتاج من كل جهة تزويدنا بالتالي:

1. من فريق إيجابي: APIs قراءة فقط لبيانات المواد، الأكواد، الأوصاف، الأسعار، التصنيفات، الكميات المتوفرة، المستودعات، الصور، وآخر تحديث، مع توثيق المصادقة وبيئات الاختبار والإنتاج.
2. من فريق Website زها: آلية إرسال طلبات الشوب أو طلبات التصنيع إلى نظام Zaha-core-orbit، والحقول المرسلة في الطلب، وآلية استقبال تحديثات حالة الطلب وتاريخ التسليم.
3. من فريق الدفع الإلكتروني: تفاصيل مخرجات عملية الدفع، أرقام المراجع، حالات الدفع، Webhooks، آلية التحقق من التوقيع، وربط عملية الدفع برقم طلب الموقع.

نقترح عقد جلسة تقنية قصيرة لتثبيت أسماء الحقول، أمثلة JSON، صلاحيات الوصول، وبيئة الاختبار، وبعدها يتم اعتماد Contract نهائي لكل API قبل بدء الربط.
