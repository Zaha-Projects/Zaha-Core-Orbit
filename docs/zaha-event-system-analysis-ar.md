# تحليل وفهم وثيقة نظام إدارة الفعاليات في مراكز زها

## 1) الهدف التشغيلي للنظام
النظام المقترح هو منصة مركزية لإدارة دورة حياة الفعالية كاملة عبر ثلاث طبقات مترابطة:
1. الأجندة السنوية
2. الخطة الشهرية لكل فرع
3. تنفيذ الفعالية والمتابعة والتقييم

الغاية الأساسية: توحيد الإجراءات بين الفروع والمركز الرئيسي (خلدا)، وتحسين المتابعة، ورفع جودة القياس والتقييم.

## 2) نموذج الحوكمة والأدوار
- **على مستوى الفروع**: relations_officer / relations_manager + امتدادات برامج بحسب الحاجة.
- **على مستوى خلدا (HQ)**: relations/programs officers & managers + executive_manager + followup_officer + communication_head + workshops_secretary.
- الوثيقة تؤكد أن **المنفذ الرئيسي لإدخال الفعالية هو relations_officer في الفرع**، بينما تدخل البرامج فقط عند `requires_programs = true`.

## 3) منطق التخطيط: سنوي → شهري → تنفيذ
### الأجندة السنوية
- تُبنى نهاية السنة للسنة التالية.
- تحتوي: الاسم، التاريخ، اليوم، النوع، (إجباري/اختياري)، (موحدة/غير موحدة).
- الفعاليات الاختيارية تسمح للفرع بتحديد المشاركة من عدمها.
- توجد مرونة تنفيذ بتاريخ ± أسبوع (كمتطلب أعمال).

### الخطة الشهرية
- تُعبأ نهاية كل شهر للشهر التالي.
- تربط فرعياً بفعاليات الأجندة، مع تاريخ تنفيذ مقترح وجهة مسؤولة (علاقات/برامج).
- تدعم أيضاً فعاليات خارج الأجندة (لكن الوثيقة تشير أن مسارها ما زال غير واضح كـ workflow مستقل).

## 4) نطاق بيانات الفعالية المطلوب
الوثيقة تحدد نموذج بيانات غني يشمل:
- معلومات عامة (اسم/وصف/نوع).
- المكان (داخل المركز: قاعة/موقع داخلي، خارج المركز: اسم مكان + خرائط).
- الزمن (من/إلى).
- الفئة المستهدفة (Lookup Data + خيار "أخرى").
- الحضور (متوقع/فعلي).
- المتطوعون (احتياج + عدد).
- فريق العمل (فرق/أعضاء/مسؤوليات).
- الشركاء والرعاة.
- المخاطبات الرسمية.
- متطلبات المشاغل.
- التغطية الإعلامية ورفع المواد بعد التنفيذ.
- المستلزمات وحالتها (متوفر/يلزم توفير).

## 5) سير الموافقات المستهدف
المسار الأساسي:
1. relations_officer (Branch) إدخال
2. relations_manager (Branch) اعتماد/إرجاع
3. HQ relations officer
4. HQ relations manager
5. executive_manager (اختياري/نهائي حسب الحالة)

ويوجد مسار برامج إضافي عندما `requires_programs = true`:
- programs_officer → programs_manager.

ملاحظة حاسمة: الورشة والاتصال **ليسا مسار اعتماد** بل مسار **ملاحظات**.

## 6) قراءة حالة النظام الحالية (كما في النص)
### منجز/قوي
- نواة الأجندة السنوية موجودة.
- الربط مع الخطة الشهرية موجود.
- مسارات موافقات متعددة المراحل موجودة.
- سجلات قرارات وتقارير/KPI أساسية موجودة.

### منجز جزئياً / فجوات
- نظام التقييم الديناميكي (Questions/Responses عبر Lookup) غير مكتمل.
- إدارة الحضور والمتطوعين غير مكتملة على مستوى الواجهات/النمذجة.
- بعض الحقول ما زالت نصية بدل Lookup Data.
- نمذجة المكان غير مكتملة بالتفصيل المطلوب.
- لا يوجد عرض واضح لعدد طلبات التعديل وتفاصيلها في كل مرحلة.
- لا توجد قواعد تحقق صريحة لبعض متطلبات الأعمال الزمنية (نهاية السنة/نهاية الشهر/± أسبوع).

## 7) ملاحظات تقنية محددة واردة في النص
- يوجد خطر برمجي محتمل في `MonthlyActivitiesController@update` لاستخدام المتغير `conflictWarning` بدون تعريف.
- بعض النماذج تتطلب تحويلات Schema/DB لتحسين جودة التقارير (Lookup-first design).

## 8) ترجمة المتطلبات إلى Backlog تنفيذي مقترح
1. **Dynamic Evaluation Module**
   - جداول: questions, question_scales/options, activity_evaluations, responses.
   - شاشة تقييم post-execution حسب دور followup_officer.
2. **Attendance & Volunteers Module**
   - expected_attendance / actual_attendance / needs_volunteers / volunteer_count.
3. **Structured Location & Target Groups**
   - `location_type`, `hall_id` أو `internal_location`, `external_place_name`, `maps_url`, `target_group_id` + custom fallback.
4. **Approval Revision Transparency**
   - عداد تعديلات per stage + سجل تفصيلي (requested change, by whom, when, resolved_at).
5. **Critical Bug Fix + Tests**
   - إصلاح `conflictWarning` وإضافة اختبارات feature/unit لمسارات update/approval.

## 9) Test Flow عملي معتمد من النص
التدفق الأكثر اتساقاً مع المتطلبات:
1. relations_officer (Branch) ينشئ الفعالية ويحدد requires_*.
2. relations_manager (Branch) يراجع ويعتمد أو يعيد للتعديل.
3. (شرطي) programs_officer ثم programs_manager إن كانت الفعالية تتطلب برامج.
4. workshops_secretary + communication_head يضيفان ملاحظات تشغيلية.
5. executive_manager اعتماد نهائي.
6. followup_officer إدخال التقييم بعد التنفيذ.

## 10) الخلاصة التنفيذية
الوثيقة تصف نظاماً ناضجاً من ناحية الفكرة المعمارية والـ workflow العام، لكنه يحتاج إغلاق فجوات "طبقة التنفيذ الفعلية" (التقييم، الحضور/المتطوعون، توحيد lookup، شفافية طلبات التعديل) للوصول إلى دورة متكاملة بالكامل: **Agenda → Monthly Plan → Execution → Evaluation**.
